# Phase 2.3: Project Organization

## Summary

Add multi-composition projects, named bookmarks, recent projects list, auto-save with recovery, and version history leveraging DSL diffability.

**Effort estimate:** Large (1-2 weeks)

---

## 1. Multi-Composition Projects

### Data Model
- `ProjectDocV2` contains `compositions[]`, each with its own graph, UI state, settings, and history reference
- One active composition at a time (engine processes one graph)
- Compositions share a project-scoped asset registry

### Composition Switching
1. Flush current composition snapshot (graph state + UI state)
2. Set active composition ID
3. Load new composition's graph into engine
4. Swap undo/redo stacks (keyed by `compositionId` in `kernel.ts`)
5. Recompute viewer; update UI selection/viewport

### Engine Handling
- Engine receives a fresh graph on switch (clear current, load new)
- Cache: clear evaluator cache on switch (compositions are independent)
- Future optimization: keep warm caches for recently-active compositions

### Composition-Level vs Project-Level Settings
- **Project-level**: color management config, asset registry, bookmarks
- **Composition-level**: graph, viewport state, selection, undo stack

## 2. Project File Format

### Desktop Directory Structure (`MyProject.cascade/`)
```
project.json                           # Manifest, schema version, composition list, asset manifest, bookmarks, settings
compositions/
  <compositionId>.dsl                  # Canonical graph (DSL text)
  <compositionId>.ui.json              # Viewport, selection, collapsed groups
assets/
  <assetId>.<ext>                      # Content-addressed assets (or sha256/<hash>)
history/
  <compositionId>/
    <versionId>.dsl                    # Snapshots (prune policy applies)
thumbs/
  <compositionId>.webp                 # Per-composition thumbnail
  project.webp                         # Project thumbnail
.lock                                  # Desktop concurrency lock
```

### Single-File Export (Browser-Friendly)
```json
{
  "schemaVersion": 2,
  "meta": { "id": "uuid", "name": "My Project", "createdAt": "...", "modifiedAt": "...", "appVersion": "x.y.z" },
  "settings": { "colorManagement": {} },
  "assets": [
    { "assetId": "a1", "contentHash": "sha256:...", "storedAs": "embedded|external", "mime": "image/png" }
  ],
  "compositions": [
    {
      "compositionId": "c1",
      "name": "Comp 01",
      "settings": {},
      "graph": { "dsl": "..." },
      "ui": { "viewport": { "x": 0, "y": 0, "zoom": 1 }, "selection": [], "bookmarks": [] }
    }
  ],
  "bookmarks": [
    { "id": "b1", "type": "node|group|preset|view", "compositionId": "c1", "label": "Key", "target": {} }
  ]
}
```

### Backward Compatibility
- Detect legacy single-graph JSON by absence of `schemaVersion: 2`
- Auto-convert to one-composition `ProjectDocV2` with `migration.from = "v1"`

## 3. ProjectStorage Interface

### Unified Persistence Adapter
```typescript
interface ProjectStorage {
  load(projectId: string): Promise<ProjectDocV2>;
  save(project: ProjectDocV2): Promise<void>;
  
  // Autosave
  writeAutosave(compositionId: string, snapshot: CompositionSnapshot): Promise<void>;
  readAutosave(compositionId: string): Promise<CompositionSnapshot | null>;
  
  // Version history
  appendVersion(compositionId: string, version: VersionEntry): Promise<void>;
  listVersions(compositionId: string): Promise<VersionEntry[]>;
  getVersion(compositionId: string, versionId: string): Promise<string>; // DSL text
  
  // Recents
  listRecents(): Promise<RecentProject[]>;
  updateRecent(project: RecentProject): Promise<void>;
  clearRecents(): Promise<void>;
  
  // Thumbnails
  putThumbnail(id: string, data: Blob): Promise<void>;
  getThumbnail(id: string): Promise<Blob | null>;
}
```

### Implementations
- `IndexedDbProjectStorage` — browser (Web)
- `TauriFsProjectStorage` — desktop (Tauri filesystem)

### Atomicity
- Desktop: write to temp file, then rename (atomic)
- Browser: write new IndexedDB record, then flip `currentHead` pointer

## 4. Bookmarks & Favorites

### What Can Be Bookmarked
- Nodes (by ID + composition)
- Groups (by group ID + composition)
- Parameter presets (snapshot of node params)
- Graph positions (viewport coordinates for quick navigation)

### Storage
- Stored in `ProjectDocV2.bookmarks[]`

### UI
- Bookmarks section in the Project sidebar
- Search/filter by label and type
- Click to navigate: select node, focus viewport, or apply param preset
- Context menu: rename, delete, update (re-save current state)

## 5. Recent Projects List

### Storage
- **Browser**: `localStorage` key `cascade_recent_projects` (JSON array of metadata)
- **Desktop**: app data dir `recents.json`
- Maximum entries: 20 (oldest evicted)

### UI

- "Open Recent" menu item in File menu

## 6. Auto-Save & Recovery

### Triggers
- **Debounced on-change**: 2 seconds after last modification
- **Interval**: every 30 seconds regardless (safety net)
- **Before navigation**: `beforeunload` handler (browser)

### Storage
- Separate from user saves (different storage channel in `ProjectStorage`)
- Desktop: temp dir within project `.cascade/autosave/`
- Browser: IndexedDB `autosave` object store

### Recovery Flow
1. On app start: check if autosave exists and is newer than last explicit save
2. If yes: show prompt "Recovered unsaved changes from [timestamp]. Restore? Compare? Discard?"
3. Restore: load autosave state into project
4. Compare: show diff view (DSL text diff)
5. Discard: delete autosave, load last explicit save

### Configuration
- User can disable autosave in settings
- Configurable interval (15s, 30s, 60s, 120s)
- Maximum autosave history: 5 entries per composition

## 7. Version History

### What Constitutes a "Version"
- **Explicit save** by user (always creates a version)
- **Auto-checkpoint** every 5 minutes of active editing (configurable)
- Each version stores: full DSL text, timestamp, reason (user_save/auto), composition ID

### Viewing History
- Version history panel in the Project sidebar
- List of versions with timestamp, type badge (manual/auto), relative time
- Click to preview: show DSL diff against current state

### Diffing
- Line-based text diff on DSL representations (no new dependencies needed — use simple LCS diff)
- Display as side-by-side or unified diff with added/removed highlighting
- DSL is designed to be human-readable and diffable

### Restoring
- "Restore this version" loads the DSL snapshot into the active composition
- Creates an undo boundary so the restoration itself is undoable
- Previous state becomes a new version (no data loss)

### Storage Efficiency
- Start with full DSL snapshots per version (simple, reliable)
- Pruning: keep all versions from last 24 hours; keep daily snapshots for last 30 days; prune older
- Future optimization: incremental diffs (deferred — full snapshots are small for most graphs)

## 8. Browser vs Desktop Differences

### Storage Limits
- **IndexedDB**: typically 50% of free disk space (browser-controlled); quota errors must be handled
- **Desktop filesystem**: limited only by disk space

### Graceful Degradation
- Quota exceeded: show warning "Storage full — some features may not work. Clear old projects?"
- Fallback: disable autosave if storage full; allow explicit saves only

### File System Access API (Browser)
- Use for "Save As" dialogs in browsers that support it
- Fallback to download-as-file for unsupported browsers

## 10. Migration & Backward Compatibility

### Opening Old Projects
- Detect `schemaVersion: 1` (or absent) → auto-migrate to `ProjectDocV2`
- Single graph becomes one composition named "Untitled"
- Migration is non-destructive: keep original file, write new format alongside (desktop)

### Forward Compatibility
- New projects in old Cascade versions: show "This project requires a newer version of Cascade"
- Unknown fields in `ProjectDocV2` are ignored (forward-compatible)

## 11. Error Handling

| Failure Mode | Response |
|---|---|
| Corrupt project file | Attempt partial load; show what could be recovered; offer "load empty" |
| Storage quota exceeded | Warning toast; disable autosave; allow explicit save to external file |
| Concurrent access (multi-tab) | Advisory lock (browser: localStorage flag; desktop: `.lock` file); show "Project open in another tab/window" |
| File locking (desktop) | Lock on open; release on close; stale lock detection (check PID if possible) |
| Partial write recovery | Atomic writes prevent corruption; if detected, fall back to last autosave |
