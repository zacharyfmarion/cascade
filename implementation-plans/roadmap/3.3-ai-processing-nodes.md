# Phase 3.3: AI Processing Nodes

## Summary

**Commodity feature, lower priority** — nice-to-have but not a differentiator.

Expand AI processing nodes (style transfer, super-resolution, background removal, color grading suggestions), support multiple providers beyond Replicate (including local ONNX), and position as optional power-ups with graceful degradation.

**Effort estimate:** Large (3d+) — mostly async queue/cancel/progress, cache persistence, and ONNX model management

---

## 1. Provider Abstraction

### Core Types
```rust
// cascade-core

#[derive(Clone, Copy, Debug, Eq, PartialEq, Hash)]
pub enum AiOp {
    Depth, Inpaint,
    StyleTransfer, SuperResolution, BackgroundRemoval, ColorGradeSuggest,
}

#[async_trait::async_trait]
pub trait AiProvider: Send + Sync {
    fn id(&self) -> &'static str;
    fn capabilities(&self) -> AiCapabilities;
    fn estimate_cost(&self, req: &AiRequest) -> Option<f64>;
    async fn run(
        &self,
        req: AiRequest,
        ctx: AiRunContext, // cancellation, progress callback, timeouts
    ) -> Result<(AiResult, AiRunMetadata), CascadeError>;
}

pub struct AiCapabilities {
    pub supported_ops: Vec<AiOp>,
    pub models: HashMap<AiOp, Vec<ModelInfo>>,
    pub supports_streaming: bool,
}
```

### Provider Selection Policy
1. If node param `provider_id` set → use it (error + degrade if unsupported)
2. Else → `default_provider_for(AiOp)` then fall back to first provider that `supports(op)`

### Provider Registry
- Engine maintains a `Vec<Arc<dyn AiProvider>>`
- Providers registered at startup based on configuration
- Engine exposes `list_ai_providers()` and `list_ai_models(op, provider)` for UI dropdowns

## 2. New AI Operations

### Style Transfer
- **Inputs**: content `Image` + style `Image`
- **Output**: stylized `Image`
- **Params**: `strength` (0-1), `style_weight` (0-1), `preserve_content` (0-1), `model_id`, `seed`
- **Fallback**: return content image unchanged + `aiNodeStale`

### Super-Resolution
- **Input**: `Image`
- **Output**: upscaled `Image`
- **Params**: `scale` (2 or 4), `quality` (0-100, maps to provider settings), `model_id`
- **Fallback**: return input at original scale + stale marker

### Background Removal
- **Input**: `Image`
- **Output**: `RGBA Image` with alpha cutout; optional separate `matte` output
- **Params**: `edge_quality` (0-100), `return_matte` (bool), `model_id`
- **Fallback**: return input with alpha = 1.0 + stale marker

### Color Grading Suggestions
- **Input**: `Image`
- **Output**: structured parameter suggestions (NOT an image)
```rust
AiResult::Params(serde_json::Value)
// e.g., { "exposure": 0.3, "contrast": 1.2, "saturation": 0.9, "temperature": 5500, "tint": -5 }
```
- **UX**: Dedicated node UI panel shows suggestions + "Apply to selected Grade node" button
- **Fallback**: empty params + stale marker

## 3. Local Model Support (ONNX)

### Desktop-First Approach
- `OnnxProvider` behind feature flag in native runtime/Tauri
- Uses `ort` (ONNX Runtime Rust bindings) for inference
- CPU inference first; optional GPU acceleration later

### Model Management
```rust
pub struct ModelManifest {
    pub name: String,
    pub version: String,
    pub url: String,      // Download URL
    pub sha256: String,   // Integrity check
    pub size_bytes: u64,
    pub ops: Vec<AiOp>,   // What operations this model supports
}
```
- Disk cache: `~/.cascade/models/`
- Download on first use with progress indicator
- LRU eviction with configurable size quota (default 5GB)

### WASM Compatibility
- ONNX Runtime does NOT run in WASM (too large, no GPU access)
- In browser: `OnnxProvider` is simply absent; nodes degrade cleanly
- Future: possible ONNX Web (onnxruntime-web) but deferred

### Memory Management
- Load model on demand, unload after idle timeout
- Only one model loaded at a time (swap if different operation needed)

### UI Separation
- AI nodes in separate "AI Power-ups" category in node library
- Distinct visual styling: subtle AI badge on node

### No-Provider Experience

  - AI nodes visible but show "Configure AI provider" CTA

  - Zero friction for non-AI users (no nag screens, no blocking)

### Feature Gating
- AI nodes always available in library (discoverable)
- Attempting to use without provider → friendly inline prompt, not an error
- Never block non-AI workflows

## 5. Graceful Degradation

### Error → Fallback Matrix

| Condition | Behavior |
|---|---|
| API unavailable | Show last cached result if available; else operation-specific fallback output |
| Invalid API key | Per-node error: "Invalid key — update in Settings → AI"; disable auto-rerun |
| Rate limited | Per-node status: "Rate limited, retry in {n}s"; auto-retry with backoff |
| Model deprecated | Per-node warning: "Model deprecated — select alternative"; suggest replacement |
| Network offline | Per-node error: "Offline — using cached result" (if available) or fallback |

### Status Indicators
- Node badge: ✓ success / ⏳ processing / ⚠️ stale/cached / ✗ error
- `aiSlice.aiNodeStatuses` already tracks per-node status
- Add `aiNodeStale` flag when using cached/fallback result

## 6. Async Execution Model

### Queue Management
- `AiExecutor` owned by `EvalSession` with bounded concurrency (default: 2 concurrent requests)
- Per-provider rate limiting (configurable)
- Queue: FIFO with priority override for user-initiated re-runs

### Progress Reporting
- Replicate provides polling progress → emit to `aiSlice` as percentage
- UI: progress bar on node, "Processing (45%)..."

### Cancellation
- Each AI request has a cancellation token
- Cancel button on node UI
- Cancel on node deletion or graph clear

### Rerun Policy
- Default: **manual rerun only** (prevent accidental cost)
- "Auto-rerun" toggle per node (opt-in)
- Rerun triggers: input image changes + auto-rerun enabled, or user clicks "Rerun"

### Timeout
- Configurable per-provider (default: 120s for cloud APIs, 300s for local ONNX)
- On timeout: cancel request, show error, offer retry

## 7. Cost Management

### Cost Estimation
- `AiProvider.estimate_cost(req)` returns estimated USD before execution
- Display in node UI: "Estimated cost: ~$0.02"

### Per-Node Cost Tracking
```typescript
interface AiRunMetadata {
    provider_id: string;
    model_id: string;
    cost_estimate_usd: number | null;
    cost_actual_usd: number | null;
    cache_hit: boolean;
}
```
- Session totals displayed in AI settings or status bar
- Per-node cost visible in inspector

### Budget Warnings
- Optional budget limit: warn when session cost exceeds threshold
- Batch rendering with AI nodes: show projected total cost before starting ("This batch will run 100 AI operations, estimated cost: ~$2.00. Proceed?")

### Avoiding Accidental Reruns
- Default to manual rerun (not auto)
- Cache aggressively: same input + same params = cached result
- "Are you sure?" for operations over cost threshold

## 8. Result Caching

### Cache Key Design
```rust
fn cache_key(req: &AiRequest) -> CacheKey {
    // Includes: provider_id, model_id, model_version,
    // input_image_hash, all normalized params (rounded floats, sorted)
    // Excludes: timestamps, request IDs
}
```

### Cache Persistence
- **Desktop**: filesystem cache in `~/.cascade/ai-cache/`
- **Browser**: IndexedDB with quota + LRU eviction

### Cache Invalidation
- Explicit: user clicks "Clear AI cache" in settings
- Model version change: different model version = different cache key
- Size-based: evict oldest entries when quota exceeded

### Storage Implications
- AI outputs can be large (upscaled images)
- Default quota: 1GB desktop, 500MB browser
- Show cache size in settings with "Clear" button

### Error Surfacing
- All AI errors propagate through existing `CascadeError` → `EngineError` pipeline
- Per-node error attribution with AI-specific error messages
- Toast notifications for session-level issues (API key invalid, provider down)
- Content filtering: clear message about why content was rejected

## 11. Testing Strategy

### Mock Provider
- Deterministic outputs for each operation
- Configurable delays, errors, and progress events
- Use for unit + integration tests in `cascade-core` and node crates

### Golden Cache-Key Tests
- Assert stable key generation across default params, float rounding, seed presence
- Verify cache hits when expected, misses when params differ

### Degradation Tests
- Simulate each `AiErrorKind` → verify: graph continues, stale flags set, cached output reused, UI status updated

### Real API Tests (Opt-In)
- Behind env vars (`REPLICATE_API_KEY`, etc.)
- Record request IDs; assert minimal invariants (non-empty image, cost metadata present)
- Run in CI only with secrets configured

### ONNX Performance Smoke
- Benchmark single model inference on desktop
- Assert no unbounded memory growth
- Verify model cache eviction logic
