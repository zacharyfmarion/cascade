## 1. Node Deletion State Cleanup

### Problem
When nodes are deleted, orphaned entries remain in state maps: `renderResults`, `nodeTimings`, `aiNodeStatuses`, `aiNodeStale`, and potentially other node-keyed maps across Zustand slices.

### Solution: `cleanupNodeState(nodeId)` Helper

```typescript
// In a shared utility or kernel.ts
function cleanupNodeState(nodeId: string) {
  // Clear from all node-keyed maps
  store.getState().renderResults.delete(nodeId);
  store.getState().nodeTimings.delete(nodeId);
  store.getState().aiNodeStatuses.delete(nodeId);
  store.getState().aiNodeStale.delete(nodeId);
  // Clear any pending render jobs for this node
  // Clear selection if deleted node was selected
  // Clear any error state for this node
}
```

### All Deletion Paths That Must Call It
1. **Direct delete**: User deletes node(s) via keyboard/context menu вҶ’ `graphSlice.deleteNodes()`
2. **Undo of creation**: Undo stack pops a node creation вҶ’ must cleanup
3. **Clear graph**: "New project" or "Load project" вҶ’ cleanup all nodes
4. **Group operations**: Node absorbed into group вҶ’ cleanup from outer graph state
5. **Template/import replacement**: Loading a template replaces the graph вҶ’ cleanup old nodes

### Race Conditions
- In-flight render for a deleted node: render result arrives after deletion вҶ’ check if node still exists before storing result
- AI operation in progress for deleted node: cancel the operation and discard result

### Key Files
- `apps/web/src/store/graphStore/slices/graphSlice.ts` вҖ” node deletion actions
- `apps/web/src/store/graphStore/slices/renderSlice.ts` вҖ” renderResults, nodeTimings
- `apps/web/src/store/graphStore/slices/aiSlice.ts` вҖ” aiNodeStatuses, aiNodeStale
- `apps/web/src/kernel.ts` вҖ” shared mutable state

### Message Protocol

```typescript
// Main вҶ’ Worker
type WorkerRequest =
  | { type: 'init', wasmUrl: string }
  | { type: 'loadGraph', graphJson: string }
  | { type: 'setParam', nodeId: string, paramId: string, value: unknown }
  | { type: 'render', requestId: string, viewerNodeId: string }
  | { type: 'cancelRender', requestId: string }
  | { type: 'loadImageData', nodeId: string, data: ArrayBuffer } // Transferable

// Worker вҶ’ Main
type WorkerResponse =
  | { type: 'initialized' }
  | { type: 'renderComplete', requestId: string, imageData: ArrayBuffer, width: number, height: number }
  | { type: 'renderProgress', requestId: string, progress: number }
  | { type: 'renderError', requestId: string, error: EngineErrorDto }
  | { type: 'renderCancelled', requestId: string }
```

### Moving the Engine to a Worker
- WASM module can be instantiated in a Worker via `importScripts` or dynamic import
- Image data transferred via `Transferable` (zero-copy ArrayBuffer transfer)
- Engine state lives entirely in the worker; main thread holds a proxy

### Render Request/Response Lifecycle
1. Main thread sends `render` request with `requestId`
2. Worker evaluates graph, produces image
3. Worker sends `renderComplete` with image ArrayBuffer (transferred, not copied)
4. Main thread receives result, updates renderResults in store, triggers viewer re-render

### Cancellation
- Main thread sends `cancelRender` with `requestId`
- Worker checks cancellation flag between node evaluations
- If cancelled, sends `renderCancelled` response

### Progress Reporting
- Worker emits `renderProgress` during evaluation (per-node completion)
- Main thread updates a progress indicator in the viewer

### Error Propagation
- Worker catches evaluation errors вҶ’ serializes as `EngineErrorDto` вҶ’ sends to main thread
- Main thread processes errors through existing error handling pipeline

### Implementation Phases
1. **Phase A**: Create Worker wrapper with message protocol; move engine init to worker; verify basic render works
2. **Phase B**: Add cancellation and progress; handle concurrent render requests (queue or cancel-and-replace)
3. **Phase C**: Optimize data transfer (SharedArrayBuffer for preview if available); handle worker crashes with auto-restart

### Key Files
- New: `apps/web/src/engine/engineWorker.ts` вҖ” Worker script
- New: `apps/web/src/engine/workerBridge.ts` вҖ” Main thread proxy implementing EngineBridge
- `apps/web/src/engine/bridge.ts` вҖ” EngineBridge interface
- `apps/web/src/store/graphStore/slices/renderSlice.ts` вҖ” render orchestration

## 3. EngineBridge Refactoring

### Current Problem
- `EngineBridge` has methods that return `Promise<T> | T` depending on backend (WASM sync vs Tauri async)
- 44+ files import from the bridge
- Feature checks scattered through code

### Proposed Split
```typescript
// CoreBridge вҖ” all methods async, works everywhere
interface CoreBridge {
  loadGraph(json: string): Promise<void>;
  evaluate(viewerId: string): Promise<EvalResult>;
  setParam(nodeId: string, paramId: string, value: unknown): Promise<void>;
  // ... all graph operations
}

// DesktopBridge extends CoreBridge вҖ” desktop-only capabilities
interface DesktopBridge extends CoreBridge {
  openFileDialog(options: FileDialogOptions): Promise<string | null>;
  readFile(path: string): Promise<ArrayBuffer>;
  writeFile(path: string, data: ArrayBuffer): Promise<void>;
}
```

### Making All Methods Async
- Convert `WasmEngine` methods to return `Promise.resolve(syncResult)` for consistency
- This unblocks the Web Worker migration (worker communication is inherently async)

### Migration Path
1. Define new async interfaces
2. Wrap existing implementations to match new interface
3. Update importing files incrementally (can be done file-by-file)
4. Remove old sync-or-async union types

### Capability Queries
```typescript
interface CoreBridge {
  capabilities(): BridgeCapabilities;
}
interface BridgeCapabilities {
  hasFileSystem: boolean;
  hasGpu: boolean;
  hasDesktopFeatures: boolean;
}
```

## 5. Performance Optimization Opportunities

### React Render Optimization
- Audit component re-renders using React DevTools profiler
- Add `React.memo` to pure display components (node views, parameter widgets)
- Use stable selector functions for Zustand subscriptions to prevent unnecessary re-renders

### Store Selector Optimization
```typescript
// Bad: creates new object every render
const { nodes, edges } = useGraphStore(state => ({ nodes: state.nodes, edges: state.edges }));

// Good: stable individual selectors
const nodes = useGraphStore(state => state.nodes);
const edges = useGraphStore(state => state.edges);
```

### xyflow Optimization
- Use `nodesDraggable`, `nodesConnectable` etc. at the flow level rather than per-node
- Implement `shouldNodeUpdate` on custom node components for fine-grained control

## 7. Error Handling

| Failure Mode | Response |
|---|---|
| Worker crash | Auto-restart worker; show "Render engine restarted" toast; re-send current graph |
| WASM OOM in worker | Surface as EngineError; suggest reducing image size or closing other tabs |
| Render timeout | Cancel after configurable timeout (30s default); show "Render timed out" with retry option |
| Stale state after failed operation | Detect inconsistency between store and engine state; offer "Sync to engine" action |
