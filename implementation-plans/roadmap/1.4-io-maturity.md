# Phase 1.4: I/O Maturity

## Summary

Complete EXR multi-layer support, improve image sequence I/O, enhance drag-and-drop, add metadata preservation, and audit common format support (WebP, AVIF, HDR).

**Effort estimate:** Medium (1-2 days) if deferring `ValueType::Channels` to Phase 6.4; Large (3d+) if implementing multi-channel values now.

**Recommendation:** Defer `ValueType::Channels` — keep current "one RGBA Image per layer output" for Phase 1.4.

---

## Current State (Already Implemented)

The codebase already has substantial EXR multi-layer infrastructure:
- `crates/cascade-core/src/exr.rs`: EXR header parsing (`parse_exr_metadata`), port-name sanitization, layer classification, `decode_all_layers`, `encode_multilayer_exr`
- `LoadImage` already adapts `spec()` dynamically based on parsed EXR metadata, adds per-layer output ports
- `LoadImageSequence` supports per-frame EXR with dynamic ports based on first-frame metadata
- Pattern utilities exist: `{frame}` / `{frame:N}` normalization, regex building
- Frontend `sequenceFrameManager.ts` handles byte caching and pattern inference
- Graph supports dynamic ports via `InstanceAwareSpecProvider` and `prune_connections_for_node`

## 2. Image Sequence I/O

### Pattern Syntax Unification
- Support `####`, `%04d`, `{frame}`, `{frame:4}` across both Rust and TS
- Normalize all patterns to internal `{frame:N}` format in Rust
- Add `parse_pattern(path: &str) -> Result<SequencePattern, CascadeError>` that detects any format

### Frame Range Parameters
- Add params to LoadImageSequence: `first_frame`, `last_frame`, `frame_offset`
- Out-of-range behavior param: `Hold` (clamp to nearest), `Black` (return transparent), `Error` (propagate error)
- Integration: evaluator passes `FrameTime.frame + frame_offset` to sequence node; node looks up file by computed frame number

### Missing Frame Handling
- Detect gaps in frame sequence (e.g., frames 1-10 exist but 5 is missing)
- Store available frames in `BTreeSet<u32>` in `SequenceInfo` for gap detection
- On missing frame: respect out-of-range behavior setting (Hold nearest / Black / Error)
- Add `CascadeError::MissingFrame { frame, path }` variant

### Cache Behavior
- Current: both Rust and TS cache 32 frames (LRU)
- Keep this default; add configurable `frame_cache_size` parameter
- Cache keyed by `(node_id, frame_number)` — already implemented

### Memory Management
- Sequence cache respects evaluator's global 512MB cache budget
- When budget exceeded, evict oldest frames first (LRU)
- For large EXR sequences: only decode requested layers per frame

### Pre-loading / Look-ahead
- Desktop (Tauri): synchronous file reads; no prefetch needed for now
- Browser: `sequenceFrameManager` can prefetch N adjacent frames in background via `file.arrayBuffer()` in idle callbacks
- Keep prefetch conservative (±2 frames) to avoid memory pressure

### Key Files
- `crates/cascade-nodes-std/src/input.rs` — LoadImageSequence evaluate
- `apps/web/src/engine/sequenceFrameManager.ts` — frame byte cache
- `apps/web/src/store/graphStore/slices/sequenceVideoSlice.ts` — sequence state

## 3. Drag-and-Drop Improvements

### Current Behavior
- `NodeCanvas.tsx` `onDrop`: creates multiple `LoadImage` nodes for dropped image files
- No sequence grouping, no folder support

### Proposed Changes

**Sequence Detection on Multi-File Drop**
- When multiple files dropped, analyze filenames for numeric patterns (e.g., `plate.0001.exr`, `plate.0002.exr`)
- Group files with matching stem + sequential numbers → create one `LoadImageSequence` node
- Remaining non-sequential files → create individual `LoadImage` nodes
- Use `sequenceFrameManager.inferPattern()` logic for detection

**Folder Drop**
- Desktop (Tauri): use `DataTransferItem.webkitGetAsEntry()` to traverse folder recursively
- Browser: `webkitGetAsEntry()` for folder access where supported; fallback to flat file list
- Scan folder for image files; apply same sequence detection logic
- If folder contains multiple sequences, create multiple `LoadImageSequence` nodes

**UI Feedback**
- Drop overlay: semi-transparent blue overlay on canvas during drag-over with "Drop images here" text
- Preview: show file count and detected type ("3 images" or "Sequence: 24 frames")
- Error: toast for unsupported file types ("Unsupported format: .psd")

### Key Files
- `apps/web/src/components/NodeCanvas.tsx` — onDrop handler
- `apps/web/src/engine/sequenceFrameManager.ts` — pattern inference

## 4. Metadata Preservation

### ImageMetadata Type
```rust
// Add to crates/cascade-core/src/types.rs
#[derive(Clone, Debug, Default)]
pub struct ImageMetadata {
    pub exif: Option<HashMap<String, String>>,     // EXIF tags as key-value
    pub iptc: Option<HashMap<String, String>>,     // IPTC tags
    pub xmp: Option<String>,                        // Raw XMP XML
    pub icc_profile: Option<Vec<u8>>,              // ICC profile bytes
    pub exr_headers: Option<HashMap<String, String>>, // EXR custom attributes
    pub custom: HashMap<String, String>,            // User/pipeline metadata
}
```

### Attachment to Image
- Add `pub metadata: Option<ImageMetadata>` field to `Image` struct
- Populated on load (LoadImage, LoadImageSequence)
- Preserved on export (ExportImage, ExportImageSequence, SaveExr)

### Propagation Rules
- **Default**: passthrough (downstream nodes inherit input metadata)
- **Transform nodes** (resize, crop, etc.): pass through but update relevant fields (e.g., resolution-related EXIF)
- **Merge nodes** (composite, blend): take metadata from primary (first) input
- **Strip**: explicit "Strip Metadata" node or export option to remove all metadata

### UI Inspector
- Add a "Metadata" tab in the node inspector panel
- Read-only display of key-value metadata pairs grouped by type (EXIF, IPTC, XMP, EXR)
- Visible when an image-outputting node is selected

### Key Files
- `crates/cascade-core/src/types.rs` — Image struct, new ImageMetadata type
- `crates/cascade-nodes-std/src/input.rs` — populate metadata on load
- `crates/cascade-nodes-std/src/output.rs` — write metadata on export

## 5. Format Support Audit

### WebP
- **Desktop (Rust)**: `image` crate supports WebP decode/encode (lossy + lossless)
- **Browser**: native decode via `<canvas>` + `createImageBitmap`; native encode via `canvas.toBlob('image/webp', quality)`
- **Color space**: sRGB only; convert from linear on encode, to linear on decode
- **Animation**: defer animated WebP support (complex, low priority)

### AVIF
- **Desktop (Rust)**: `ravif` crate for encode; `libavif` or `image` crate (via `avif-decode` feature) for decode
- **Browser**: native decode in most modern browsers; encode support limited (use `<canvas>` where available)
- **Color space**: supports wide gamut (P3, Rec.2020) and HDR; respect ICC profiles on decode
- **WASM**: `ravif` may not compile to WASM easily — use browser native APIs for AVIF in web mode

### HDR (Radiance .hdr)
- **Desktop + WASM**: `image` crate supports Radiance HDR decode/encode
- **Color space**: already f32 RGB; convert to RGBA (alpha = 1.0) and handle sRGB primaries assumption

### Format Detection
- Detect by file extension + magic bytes
- Fall back to `image` crate's format sniffing

### Export Settings
- Per-format quality/compression params exposed in ExportImage node
- WebP: quality (0-100), lossless toggle
- AVIF: quality, speed preset, lossless toggle
- EXR: compression method (PIZ, ZIP, ZIPS, etc.) — already supported
- PNG: compression level — already supported

## 6. Browser vs Desktop Considerations

### File System Access
- **Browser**: File input `<input type="file">`, drag-and-drop `File` objects, optional File System Access API for save
- **Desktop (Tauri)**: Full filesystem access via Tauri commands and `dialog.open`/`dialog.save`

### Large File Handling in WASM
- WASM linear memory limit: ~4GB theoretical, ~2GB practical
- For large EXR files: warn user when file exceeds threshold (e.g., 500MB)
- Selective layer decode: only decode layers connected to downstream nodes (already implemented via `decode_all_layers` with `requested_ports`)

### Streaming Decode
- Defer streaming/progressive decode to a future optimization pass
- Current approach: load full file into memory, then decode — acceptable for Phase 1.4

## 8. Performance Considerations

### Large EXR Files
- Selective layer decode already implemented (`requested_ports` parameter)
- Add "lazy decode" mode: only decode a layer when its output port is actually pulled by the evaluator
- For preview: decode at proxy resolution first (already supported via `preview_scale`)

### Sequence Memory Pressure
- Both Rust and TS caches limited to 32 frames (LRU) — configurable
- Total cache bounded by evaluator's 512MB budget
- Evict oldest frames first when budget exceeded

### Background Loading
- Desktop: synchronous file I/O is fast enough for most use cases
- Browser: file bytes already loaded asynchronously via `file.arrayBuffer()`; prefetch ±2 adjacent frames during playback

## 10. Testing Strategy

### Rust Unit Tests
- EXR multilayer: load multi-layer EXR → verify correct layers decoded → encode → verify round-trip
- Sequence patterns: test all pattern formats (`####`, `%04d`, `{frame:4}`) → verify normalization
- Missing frame handling: verify Hold/Black/Error modes
- Metadata: load image with EXIF → verify metadata populated → export → verify metadata preserved

### Frontend Tests
- Sequence grouping on drop: mock file list → verify correct sequence detection
- Pattern inference: test `sequenceFrameManager.inferPattern()` with various filename formats
- Error display: verify toast/badge rendering for I/O errors

### WASM-Specific Tests
- Large file handling: verify graceful error when file exceeds memory limit
- Format support: verify browser-native decode path works for WebP/AVIF
- Cross-browser: test on Chrome, Firefox, Safari for format support differences

### Test Fixtures
- Create small test files for each format (checked into `tests/fixtures/` or generated at test time)
- EXR: multi-layer with 2-3 named layers, single-layer, corrupt file
- Sequence: 5-frame numbered sequence with a gap, various naming patterns
