# Phase 2.4: Batch / Headless Rendering

## Summary

Build a CLI (`cascade`) on top of a public `cascade-runtime` library API for headless rendering, batch processing, graph validation, and CI/CD integration. Support parameterized rendering and GPU headless mode.

**Effort estimate:** Large (3d+)

---

## 1. CLI Design (clap)

### Binary
Single entrypoint: `cascade` (in `crates/cascade-cli/` or `crates/cascade-runtime/src/bin/cascade.rs`)

### Subcommands
```
cascade render <GRAPH> [--target <node>] [--input ...] [--output ...] [--set ...] [--params <file>]
cascade batch <GRAPH> --inputs <glob|dir> --outputs <dir> [--pattern ...] [--jobs N] [--continue-on-error] [--resume]
cascade inspect <GRAPH> [--json] [--nodes] [--params] [--ports]
cascade validate <GRAPH> [--dry-run] [--json]
```

### Core Flags (shared)
- `--format json|text` — output formatting (default `text`)
- `--json` — shorthand for `--format json`
- `--log-level error|warn|info|debug|trace` (default `info`)
- `--quiet` — only errors

### GPU/Backend Flags
- `--backend auto|cpu|gpu` (default `auto`)
- `--gpu-adapter auto|high-performance|low-power`
- `--gpu-backend auto|vulkan|metal|dx12|gl`
- `--gpu-fallback allow|deny` (default `allow` in auto)
- `--gpu-concurrency N` (default `1`)

### Exit Codes (Stable, CI-Friendly)
| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | Render/runtime error (I/O, GPU, eval, shader, OOM) |
| 2 | Validation/config error (invalid graph, bad overrides, missing nodes) |
| 3 | Internal/unexpected error |

### Stdout/Stderr Conventions
- `stdout`: primary output (result paths, JSON reports, inspect data)
- `stderr`: logs + progress (never intermixed with JSON stdout)
- Batch progress (text mode): `[12/100] Processing plate_0012.exr...`
- Batch progress (JSON mode): NDJSON events to stderr

## 2. Graph Loading & Validation

### Inputs
- JSON graph: `graph.json`
- DSL graph: `graph.cas` (detect by extension or content)
- Stdin: `cascade render -` reads from stdin

### Validation Stages
1. Parse graph (JSON/DSL)
2. Verify all node type IDs exist in registry
3. Validate ports & connections (type compatibility, no cycles)
4. Validate renderability: at least one render target exists

### Dry-Run
- `cascade validate --dry-run` runs validation + override parsing but does not initialize GPU or decode inputs

### Missing Dependencies
- Text mode: list missing node type IDs + where used
- JSON mode: `{ "missing_node_types": [{ "type_id": "...", "nodes": [...] }] }`

## 3. Input/Output Override

### Node Selectors (Single Scheme)
- `@<node_id>` — stable ID from serialization
- `name:<label>` — by node label (must be unique)
- `type:<type_id>` — by node type (error if ambiguous)

### Examples
```bash
cascade render graph.json --input @n12=plate.exr --output name:WriteMain=out.exr
cascade render graph.json --set @n12.path="plate.exr" --set name:Blur1.radius=5
```

### What `--input/--output` Does
- `--input <selector>=<path>` sets the input-path param for that node
- `--output <selector>=<path>` sets the export-path param
- If selector points to non-I/O node: error with code 2 + suggest `--set`

### Format Inference
- Output format inferred from extension (`.exr`, `.png`, `.jpg`, etc.)
- If graph hard-codes format, `--output` also sets `format=auto` unless user explicitly sets via `--set`

## 4. Batch Mode

### Command Shape
```bash
cascade batch graph.json --inputs "plates/**/*.exr" --outputs out/ --pattern "{stem}.png" --jobs 8
```

### Discovery
- Expand glob patterns; if directory, default glob: `**/*.{exr,png,jpg,jpeg,tif,tiff}`
- Sort deterministically (lexicographic) for stable CI output
- Configurable: `--include-ext`, `--exclude`

### Parallelism
- Per-file parallelism using worker pool (`--jobs N`, default = logical cores)
- GPU: gate GPU render behind semaphore (`--gpu-concurrency`, default 1) to avoid device contention

### Output Naming
- `--pattern "{stem}{suffix}.{ext}"` with variables: `{stem}`, `{ext}`, `{dir}`, `{index}`, `{hash}`
- `--suffix "_out"` convenience flag
- Collision handling: `--on-collision error|overwrite|increment` (default `error`)

### Error Handling
- `--continue-on-error`: record per-file errors; exit code 1 if any failures
- Default: fail-fast on first error

### Resume
- `--resume`: skip items where output exists
- Better: skip if sidecar `.cascade.json` matches (input mtime/hash + graph hash + overrides hash)

## 5. Parameterized Rendering

### Generic `--set`
```bash
--set <node_selector>.<param>=<value>
--set @n42.radius=5
--set name:Blur1.radius=5
```

### Type Parsing (from NodeSpec)
| Type | Syntax | Example |
|------|--------|---------|
| int | `5` | `--set @n1.count=5` |
| float | `0.25` | `--set @n1.amount=0.25` |
| bool | `true/false/1/0` | `--set @n1.enabled=true` |
| enum | variant name | `--set @n1.mode=Multiply` |
| color | `r,g,b[,a]` or `#RRGGBB` | `--set @n1.color=1.0,0.5,0.0` |
| vec2/vec3 | `x,y` / `x,y,z` | `--set @n1.offset=10,20` |

### Presets/Config Files
```yaml
# params.yaml
sets:
  - "@n1.radius=5"
  - "name:Blur.radius=3.2"
vars:
  input_dir: "/data/plates"
```
`cascade render graph.json --params params.yaml`

### Template Variables (Batch)
- `{input.path}`, `{input.stem}`, `{input.ext}`, `{output.dir}` available in string params
- Simple `{key}` replacement from a known variable map

## 6. CI/CD Integration

### Docker Images
- `cascade:cpu` — baseline, reliable, no GPU dependencies
- `cascade:gpu` — includes Vulkan/GPU libraries for GPU-enabled rendering

### GitHub Actions Example
```yaml
- name: Validate graph
  run: cascade validate graph.json --json

- name: Render batch
  run: cascade batch graph.json --inputs plates/ --outputs artifacts/ --pattern "{stem}_graded.exr" --json > report.json

- name: Upload artifacts
  uses: actions/upload-artifact@v4
  with:
    name: renders
    path: artifacts/
```

### JSON Output Schema (Stable)
```json
{
  "command": "render",
  "graph": "graph.json",
  "backend": "cpu",
  "timings": { "total_ms": 1234, "per_node": {} },
  "outputs": [{ "path": "out.exr", "size_bytes": 12345 }],
  "errors": [],
  "warnings": [{ "code": "GPU_FALLBACK", "detail": "..." }],
  "version": "0.8.0"
}
```

### Headless GPU in CI
- Default to CPU unless `--backend gpu`
- If GPU requested and adapter fails: exit 1 (unless `--gpu-fallback allow`)

## 7. Library Mode (Embedding)

### Public Rust API (`cascade-runtime`)
```rust
// Minimal public surface — CLI is a thin wrapper
RuntimeEngine::new(options: RuntimeOptions) -> Result<Self, CascadeError>
RuntimeEngine::load_graph(source: GraphSource) -> Result<LoadedGraph, CascadeError>
LoadedGraph::validate() -> Result<ValidationResult, CascadeError>
LoadedGraph::apply_overrides(overrides: Vec<Override>) -> Result<(), CascadeError>
LoadedGraph::render(target: RenderTarget, opts: RenderOpts) -> Result<RenderResult, CascadeError>
RenderResult::{image(), bytes(), write_to(path: &Path)}
```

### Thread Safety
- `RuntimeEngine` is `Send + Sync` if safe; otherwise document "one engine per worker thread"
- Return owned images/bytes; keep GPU resources internal

### Node.js Bindings (Deferred)
- Near-term: use CLI as subprocess (JSON output makes it practical)
- Future: N-API via `napi-rs` for in-process bindings

### Python Bindings
- Deferred until Rust API stabilizes

## 8. GPU in Headless Mode

### wgpu Adapter Selection
- Request adapter with `compatible_surface: None` (headless)
- `--gpu-adapter` and `--gpu-backend` flags for CI debugging

### Software Fallback
- `--backend auto`: try GPU, fall back to CPU based on `--gpu-fallback`
- Prefer CPU fallback over mystery-slow software rendering

### Messaging
- Text mode: one clear warning line on stderr
- JSON mode: `{ "warnings": [{ "code": "GPU_FALLBACK", "detail": "..." }] }`

---

## 9. Performance Considerations

### Batch Parallelism
- Default: per-file parallelism (simplest, matches typical pipeline usage)
- GPU: cap GPU concurrency; let CPU decode/encode overlap with GPU work

### Memory Management
- Stream per-item: decode → process → encode → release; don't accumulate decoded images
- `--max-inflight` flag (advanced) for explicit control

### Cache Reuse
- Reuse loaded graph + compiled shaders + node registry across batch items within a worker
- Don't attempt cross-item evaluator cache (different inputs = different outputs)

### GPU Resource Pooling
- One GPU context per process; share across workers if safe, otherwise serialize GPU renders

## 10. Error Handling

### User-Facing Categories
- Validation/config errors → exit 2
- Render/runtime errors → exit 1

### Structured JSON Error
```json
{
  "code": "SHADER_COMPILE_ERROR",
  "message": "Failed to compile GLSL for node Blur1",
  "phase": "render",
  "node": { "id": "n42", "name": "Blur1", "type": "gpu_script" },
  "param": null,
  "path": null
}
```

### Common Failures
- Missing input files → clear error with path
- Unsupported format → suggest supported alternatives
- GPU init failure → suggest `--backend cpu`
- Invalid param override → show expected type from NodeSpec
- OOM → suggest `--jobs 1` or `--backend cpu`

## 11. Testing Strategy

### CLI Integration Tests
- `assert_cmd` + `tempfile`: run `cascade validate/render/batch` on small fixture graphs
- Verify exit codes + JSON schema + deterministic output paths

### Batch Tests
- Create tiny input folder; run batch with `--resume`; assert second run skips
- Test `--continue-on-error` with one invalid input

### Override Tests
- Table-driven tests: `--set` parsing vs NodeSpec types
- Verify error messages for type mismatches

### Render Correctness
- "Known output" graphs (constant color → export) → validate by pixel values or hashes

### Headless GPU Tests
- Optional: run if GPU available, skip with clear message otherwise
- Mirror existing GPU test pattern in `cascade-gpu`
