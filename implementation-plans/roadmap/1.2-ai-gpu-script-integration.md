# Phase 1.2: AI + GPU Script Integration

## Summary

Unify the currently disconnected AI chat workflow and GPU Script node workflow so the AI can create, read, and modify GPU Script nodes through the graph DSL. Improve the GPU Script editing experience with better error messages, parameter hot-reload, and live preview.

**Effort estimate:** Large (1-2 weeks)

---

## 1. DSL Extension for GPU Script Nodes

### Proposed Syntax
```
effect = GpuScript(
  glsl: """
    vec4 process(vec4 color, vec2 uv, ivec2 pixel) {
      float intensity = params.intensity;
      return mix(color, vec4(1.0 - color.rgb, color.a), intensity);
    }
  """,
  params: {
    intensity: { type: "float", default: 0.5, min: 0.0, max: 1.0, ui: "slider" }
  }
)
```

### Multi-line GLSL Encoding
- Use triple-quote `"""..."""` for multi-line GLSL blocks in DSL text format
- Escaping: only `"""` within GLSL needs escaping (extremely rare in shader code)
- Alternative: GLSL can also be provided as a single-line escaped string for programmatic use

### Parsing
- Extend DSL parser to recognize `GpuScript` node type with `glsl` and `params` special fields
- GLSL body is passed verbatim (no DSL-level interpretation of shader code)
- Params block maps directly to KernelManifest parameter declarations

## 2. AI Tool Extensions

### New/Modified Tools

**`create_gpu_script` tool**
```json
{
  "name": "create_gpu_script",
  "input": {
    "handle": "string (variable name in DSL)",
    "display_name": "string",
    "glsl_body": "string (GLSL process() function)",
    "params": [{
      "name": "string",
      "type": "float | int | bool | vec2 | vec3 | vec4",
      "default": "number | array",
      "min": "number (optional)",
      "max": "number (optional)",
      "ui_hint": "slider | color_picker | dropdown | checkbox"
    }],
    "description": "string (optional)"
  }
}
```

**`read_gpu_script` tool**
```json
{
  "name": "read_gpu_script",
  "input": { "node_id": "string" },
  "output": {
    "glsl_body": "string",
    "params": "[ParamSpec]",
    "compile_status": "ok | error",
    "compile_errors": "[string]"
  }
}
```

**`edit_gpu_script` tool**
```json
{
  "name": "edit_gpu_script",
  "input": {
    "node_id": "string",
    "glsl_body": "string (optional - full replacement)",
    "params": "[ParamSpec] (optional - full replacement)",
    "validate_before_apply": "boolean (default true)"
  }
}
```

### Safety Constraints
- `validate_before_apply: true` compiles the GLSL before committing changes; returns errors to the AI if compilation fails so it can self-correct
- AI-generated GLSL goes through the same safety linter as user-written code

---

## 3. GLSL Read/Modify Flow

### Full Replacement (Not Diff)
- AI reads current GLSL via `read_gpu_script`, modifies locally, writes back via `edit_gpu_script`
- Full replacement avoids diff/patch complexity and GLSL-specific merge issues
- The AI sees the complete GLSL body, so it has full context for modifications

### Compilation Validation
- Before applying changes, compile the new GLSL through the naga pipeline
- If compilation fails, return structured errors (line, column, message) to the AI
- AI can self-correct based on error messages (up to 2 retries before surfacing to user)

---

## 4. Parameter Schema Generation

### AI-Determined Parameters
- AI analyzes its generated GLSL to determine `uniform` declarations
- For each uniform, AI provides: name, type, default, min/max range, UI hint, display name
- Relationship: each `uniform` in GLSL maps 1:1 to a `param` in KernelManifest

### Heuristics for Sensible Defaults
- `intensity/amount/strength/opacity` → float `[0, 1]` default 0.5, slider
- `radius/size` → float `[0, 100]` default 5.0, slider
- `angle/rotation` → float `[-180, 180]` default 0.0, slider
- `color/tint` (vec3/vec4) → color picker, default white
- `seed` → int `[0, 9999]` default 0
- `enabled/active` (bool) → checkbox, default true

### Validation
- Verify every `uniform` referenced in GLSL has a corresponding param entry
- Clamp defaults to declared ranges
- Reject invalid ranges (min >= max, NaN, inf)

## 5. Compilation Error Handling

### Better Error Messages
- When naga/wgpu compilation fails, map error spans back to user GLSL line numbers
- Maintain a `user_start_line` offset (accounting for engine GLSL prelude wrapper)
- Return structured errors: `{ line, column, message, snippet }` through the WASM bridge as `EngineError`

### Error Recovery
- On compilation failure, revert to last working GLSL (stored as `last_known_good_glsl` on the node)
- Show error overlay on the node with compilation diagnostics
- User can edit to fix, or the AI can attempt correction

### Real-time Validation During Editing
- Debounced validation (300ms after last keystroke)
- Show inline error markers in the code editor (red squiggles at error lines)
- Compilation status indicator: green checkmark / red X

---

## 6. Parameter Hot-Reload

### What Requires Recompilation
- Any change to GLSL code body
- Adding/removing parameters (changes uniform buffer layout)
- Changing parameter types

### What Can Update Without Recompilation
- Parameter value changes → update uniform buffer only (fast path)
- This is the common case for slider/color picker interaction

### Implementation
- Track a `pipeline_key` based on `(glsl_hash, param_layout_hash)`
- If key unchanged, only update uniform values
- If key changed, trigger full recompilation

## 7. Preview During Editing

### Split-View Editor
- Left pane: GLSL code editor (Monaco or CodeMirror with GLSL syntax highlighting)
- Right pane: live preview rendering the current shader

### Debounced Compilation
- Compile after 150-300ms of no typing
- Show "Compiling..." indicator during compilation
- On success: update preview with new shader
- On failure: keep showing last successful render; overlay error diagnostics

### Error Overlay
- Semi-transparent error panel over the preview area
- Shows compilation errors with line references
- Clicking an error jumps to the relevant line in the editor

## 8. End-to-End UX Flow

1. **User describes effect in chat**: "Create a chromatic aberration effect"
2. **AI generates GLSL + params**: Uses `create_gpu_script` tool with complete GLSL body and parameter schema
3. **Node appears in graph**: Connected to the appropriate input/output based on context
4. **User tweaks sliders**: Hot-reloaded uniform updates (no recompilation)
5. **User asks AI to modify**: "Make it more subtle and add a center offset parameter"
6. **AI reads current GLSL** via `read_gpu_script`, modifies, writes back via `edit_gpu_script`
7. **Preview updates**: Automatic recompilation and render after edit

### Feedback at Each Step
- Step 2: Streaming GLSL generation visible in chat (collapsible code block)
- Step 3: Node creation animated; auto-selected in graph
- Step 5-6: Chat shows "Reading shader... Modifying... Compiling..." states
- Step 7: Viewer updates automatically; before/after comparison available

---

## 9. Error Handling

| Failure Mode | Response |
|---|---|
| GLSL compilation fails | Return errors to AI for self-correction (2 retries); then surface to user |
| GPU device lost during compilation | Attempt device recreation; show "GPU unavailable" error on node |
| AI generates invalid GLSL | Compilation validation catches it before applying; AI retries |
| Parameter type mismatch | Validation rejects; clear error message about expected vs actual type |
| Shader timeout / infinite loop | wgpu device lost timeout triggers; mark shader as unsafe; revert to last good |
| AI generates params without matching uniforms | Validation catches; error returned to AI |

### DSL Round-Trip Tests

- Test multi-line GLSL encoding/decoding
- Test parameter schema preservation

### Parameter Binding Tests
- Verify uniform buffer layout matches declared params
- Verify slider changes update uniforms correctly
- Verify type validation catches mismatches
