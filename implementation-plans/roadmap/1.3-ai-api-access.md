# Phase 1.3: AI API Access (CORS)

## Summary

Resolve the remaining CORS/API access issues for AI features: add direct Replicate API calls from the Tauri Rust backend (bypassing the proxy for desktop), and add clear user-facing documentation about BYOK (bring your own key) requirements.

**Effort estimate:** Medium (1-2 days)

---

## 1. Tauri Direct Replicate Access

### Design
- Move Replicate API calls to the Rust backend for Tauri desktop builds
- Add a Tauri command: `invoke('replicate_run', { model, input, apiKey })` → returns prediction result
- Frontend detects runtime: if `window.__TAURI__` exists, use direct Tauri IPC; otherwise use Cloudflare Worker proxy

### API Key Passing
- Frontend reads key from secure storage → passes to Tauri command per-request
- Keys are never stored in Rust-side persistent state — passed transiently per call
- Tauri command validates key format before making the request

### Response Streaming
- Replicate predictions are async (poll-based): submit → poll status → get result
- Tauri command handles the polling loop internally, returning the final result
- Optionally emit progress events via Tauri events for long-running predictions

### Rust Implementation
- Add `replicate.rs` module in `apps/tauri/src-tauri/src/`
- Use `reqwest` for HTTP calls (already likely a dependency)
- Handle Replicate API response types: `starting`, `processing`, `succeeded`, `failed`, `canceled`

## 2. API Key Management

### Storage
- **Browser**: `localStorage` (simple, accessible); keys are per-provider (`anthropic_api_key`, `replicate_api_key`)
- **Tauri**: Tauri's plugin-store (encrypted at-rest on macOS Keychain / Windows Credential Manager / Linux Secret Service)

### Validation on Entry
- Format check: Anthropic keys start with `sk-ant-`, Replicate keys are UUIDs
- Optional live validation: make a lightweight API call (e.g., list models) to verify key works
- Show validation result inline (green check / red X with error message)

### Key Rotation/Update UX
- Settings panel with per-provider key fields (masked by default, reveal toggle)
- "Test" button per key
- Clear button to remove a key

### Invalid/Expired Key Handling
- API returns 401 → surface as `AuthInvalid` error on the node
- Prompt user to update key via toast with link to settings
- Do not retry with the same invalid key

## 3. Proxy Architecture Review

### Current Cloudflare Worker (`workers/proxy/worker.js`)
- Adds CORS headers for Replicate API calls from browser
- Passes through the user's API key (from Authorization header)

### Security Considerations
- The proxy is a pass-through — it does NOT store or validate keys server-side
- Rate limiting: add per-IP rate limiting in the Worker (e.g., 60 requests/minute) to prevent abuse
- The Worker should only proxy to `api.replicate.com` — reject other destinations
- Add request size limits to prevent abuse

### Deployment
- Workers are deployed via `wrangler`; document the deployment process
- Configuration: proxy URL stored in frontend env variable (`VITE_REPLICATE_PROXY_URL`)

## 5. Dual-Path Architecture

### Clean Separation
```typescript
// EngineBridge or a new AiTransport interface
interface AiTransport {
  runPrediction(model: string, input: Record<string, unknown>, apiKey: string): Promise<PredictionResult>;
}

class ProxyAiTransport implements AiTransport { /* Cloudflare Worker path */ }
class TauriAiTransport implements AiTransport { /* Direct Tauri IPC path */ }

// Selection at init
const aiTransport = isTauri() ? new TauriAiTransport() : new ProxyAiTransport();
```

### Feature Detection
- `isTauri()`: check `window.__TAURI__`
- Both transports implement the same interface — AI nodes don't know which path is used
- Anthropic API: already works in browser via `anthropic-dangerous-direct-browser-access` header; Tauri could also go direct

### Testing Both Paths
- Unit tests with mocked HTTP responses for both transports
- Integration tests: proxy path tested in dev server; Tauri path tested in Tauri dev mode

## 6. Error Handling

| Error | UX Response |
|---|---|
| Invalid API key (401) | Per-node error: "Invalid API key. Update in Settings → AI." Toast with shortcut. |
| Rate limited (429) | Per-node error: "Rate limited. Retry in {retry_after}s." Auto-retry with backoff. |
| Network failure | Per-node error: "Network error. Check connection." No auto-retry. |
| Proxy downtime | Per-node error: "AI proxy unavailable." Suggest checking back later. |
| Replicate service outage | Per-node error: "Replicate service unavailable." Link to status page. |
| Key not configured | Per-node empty state: "API key required. Set up in Settings → AI." |

### Graceful Degradation
- All AI nodes show a clear "AI unavailable" state (not a crash)
- Non-AI nodes continue working normally
- Cached AI results remain available even when API is down

## 7. Security Considerations

### Browser
- API keys in localStorage are accessible to any JS on the page — acceptable for BYOK model
- XSS mitigation: standard CSP headers, no `eval()`, sanitize all user inputs
- Keys are never included in: share URLs, template exports, graph serialization, error reports

### Cloudflare Worker
- Worker sees the key in transit (pass-through) — acceptable for a CORS proxy
- Worker does NOT log keys or request/response bodies
- Pin the Worker to only proxy Replicate API endpoints

### Tauri
- Keys stored in OS-level secure storage (Keychain/Credential Manager)
- Keys passed to Rust commands per-request, not stored in Rust process memory long-term
- No key data in Tauri logs

### General
- Never send keys to first-party servers
- Never include keys in telemetry/analytics
- Sanitize error messages to never leak key values

## 8. UX Flow: First-Time Setup

1. User clicks an AI node or opens AI chat
2. If no key configured → inline prompt: "To use AI features, add your API key in Settings"
3. User opens Settings → AI Services
4. Enters Anthropic key → clicks "Test" → sees green checkmark
5. Enters Replicate key → clicks "Test" → sees green checkmark
6. Returns to AI feature → it works immediately
7. Keys persist across sessions (localStorage / secure storage)
